COSS Protocol

Minimalist structure for efficient communication

Core Principle
Purpose: Minimalist structure for efficient communication
Why: Clear distinction between function and rationale reduces ambiguity
Tag: v3.5 semantic-clarity fluid-first flexible-keywords
Axiom
Definition: fluid first. structure when needed.
Why: Right format for the right situation
Tag: guiding-principle

Format Rules

Core syntax and structure

Basic Structure
Line: single entry
Format: `keyword. data`
Separator: `. `
Keywords
Format: single case-sensitive word/hyphenated phrase
Data: case-sensitive natural language brief phrase
Hierarchy
Indent: parent-child 0/2/4 space hierarchy
Comments: `;` for comments
Containers: `label.` for named-slugged-container
Telegraphic Format
Rule: max 5 `keyword. data` pairs per line
Separator: ` . ` between pairs
Data: no slug - use natural language

Keyword Flexibility

Adaptation for different domains

Keyword Creation
Definition: Users can create new keywords not in core set
Examples:
• Domain-specific: bug. feature. test. metric.
• Project-specific: milestone. deliverable. stakeholder.
Constraint: Must follow `keyword. data` format and indentation rules
Why: Enables domain-specific optimization while maintaining structure
Keyword Consistency
Definition: Use chosen keywords consistently within a document/system
Why: Prevents confusion within shared contexts
Example: If using `task.` instead of `action.`, use it throughout

Semantic Keywords

Distinguishing function from rationale

Purpose (pur.)
Definition: [intended function | objective | goal]
Usage: Describe what a component *does* or is *for*
Example:
lbl. error-handling pur. Structured problem reporting
Tag: functional-definition
Rationale (why.)
Definition: [rationale | justification | underlying cause]
Usage: Explain the reasoning *behind* a rule, decision, or structure
Example:
lbl. axiom def. fluid first. structure when needed. why. Right format for the right situation
Tag: justification

Hierarchical Format

Clear relationship visualization

Context Block (ctx.)
Definition: [environment | situation | cause]
Purpose: Establish broader context . Establish cognitive starting point
Why: Shared understanding prevents misalignment
Indent: 0 (top-level)
Object Block (obj.)
Definition: [primary focus | topic | concern]
Purpose: Define core message/subject . Define current cognitive target
Why: Clear focus prevents scope creep
Indent: 2
ctx. project environment
obj. specific task focus
sta. objective fact about situation
ste. subjective state or feeling
sig. next physical action

Data Blocks

Status and state information

Status Data (sta.)
Definition: [objective fact | data | status]
Purpose: State verifiable information
Why: Factual foundation enables sound decisions
Indent: 4
Tag: fact . reality-check
State Data (ste.)
Definition: [subjective state | feeling | hypothesis]
Purpose: State internal reality/intuition . Record subjective experience
Why: Emotional context informs appropriate response
Indent: 4
Tag: feeling hypothesis . self-awareness
Response Block (sig.)
Definition: [action | decision | next step]
Purpose: Define next physical action
Why: Clear next steps prevent paralysis
Indent: 2
Tag: action . activation

Inline Format

Maximum speed communication

Telegraphic Style
Purpose: Urgent situations require condensed information
Definition: ctx. [content] . obj. [content] . sta. [content] . ste. [content] . sig. [content]
Constraint: Max 5 keywords/block, one line only
Example:
ctx. team chat . obj. status update . sta. task complete . ste. on track . sig. proceed next phase
Tag: rapid-format

Error Handling

Structured problem reporting

Error Reporting
Purpose: Systematic error handling prevents issue escalation
Definition: err. [detected issue | invalid data | missing data]
Example:
ctx. data import . obj. customer records . err. missing email in 12% records
Tag: resilience

Validation Checks

Quality assurance protocols

Data Quality
Purpose: Verify information reliability
Why: Unverified data leads to flawed decisions
Check: verify origin & veracity & relevancy
Verdict: Valid | Invalid | Questionable
Logic Soundness
Purpose: Ensure reasoning integrity
Why: Sound logic prevents systemic failures
Check: reasoning & structure
Verdict: Sound | Unsound | Flawed
Execution Alignment
Purpose: Confirm action-purpose alignment
Why: Misaligned execution wastes resources
Check: right thing & thing right
Verdict: Aligned | Misaligned | Partial
